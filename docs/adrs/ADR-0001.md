## ADR-0001: defineConfig 形式と module augmentation 導入【ABANDONED】

**ステータス**: ABANDONED（2026-05-03）

**Abandon 理由（要約）**: 出発点の前提が二重に誤っていた。

1. 「user が `<PackageScripts>` generic を書く必要がある」という framing 自体が誤り。`init` の default template は `TSGitHookConfig`（generic なし）で、`<PackageScripts>` 付与はコメント例として示されているだけの opt-in。この project の type-safety の主軸は **config 構造（hook 名 / sequential / glob 形式）の型** であって、script 名の string validation ではない。
2. 仮に script-name 自動補完に価値があったとしても、実装中に `defineConfig` を value 形で使うと runtime entry が必要だと判明し、`rolldown.config.mjs` への library bundle 追加 + `package.json` `main`/`exports` 追加というスコープ拡張（→ ADR-0002 案）が必要になった。元々の改善幅に見合わない。

ソース・テスト・README は `git restore` で revert 済み。詳細は `docs/adrs/calibration.md` の 2026-05-03 エントリ参照。

**日時**: 2026-05-02T00:00:00+09:00
**タスク概要**: TypeScript-first を謳う ergonomics を強化するため、`defineConfig()` wrapper と `KnownScripts` interface の declare merge を導入し、user が generic を手書きせずに `package.json` script 名の autocomplete を受けられるようにする。

### コンテキスト

ts-git-hooks は「TypeScript First」を design philosophy の一つに掲げているが、現状の user 体験は次の点で他の TS-first tool（vite / vitest 等）に劣る:

1. `git-hooks.config.ts` で `TSGitHookConfig<PackageScripts>` の generic を user が手書きする必要がある。`init.ts:15` の default template は `TSGitHookConfig`（generic なし）なので、何もせずに開始すると autocomplete が `string` に解決されて型安全性が事実上効かない。
2. `export const config = ...` という named export 縛りで、vite/vitest の `export default defineConfig({ ... })` という業界慣行と揃わない。
3. `import type { PackageScripts } from './git-hooks.d.ts'` を user が忘れると generic が undefined になる。

これらは「TypeScript First」の宣伝に対して ergonomics が rough。`defineConfig()` 形式 + module augmentation で「`defineConfig({ ... })` を 1 個書くだけで型が効く」を実現する。

### 決定内容

次の 2 段階を **同じ PR** で実装する:

**Stage 1: `defineConfig` wrapper の追加**
- `src/index.ts` に `defineConfig()` を export
- `src/core/config.ts` の `loadConfig` を `configModule.default ?? configModule.config` の優先順で読むよう変更（後方互換のため `config` named export も継続 support）

**Stage 2: module augmentation で generic を消す**
- `src/index.ts` で空の `interface KnownScripts {}` を export
- `defineConfig` の signature が `keyof KnownScripts extends never ? string : Extract<keyof KnownScripts, string>` を解決するようにする
- `src/core/type-generator.ts` の出力を、従来の `export type PackageScripts = ...` から、`declare module 'ts-git-hooks' { interface KnownScripts { ... } }` 形式に変更
- `src/commands/init.ts` の default template を以下に変更:
  ```ts
  import { defineConfig } from 'ts-git-hooks';
  export default defineConfig({
    'pre-commit': { '*.{js,ts,jsx,tsx}': ['lint', 'test'], '*.{md,json}': 'format' },
    'pre-push': 'build',
  });
  ```

**後方互換**:
- 既存の `export const config: TSGitHookConfig<PackageScripts> = { ... }` は引き続き動く
- 既存 `git-hooks.d.ts` の `export type PackageScripts` も生成し続けるか、新形式と並存させるかは設計余地あり（推奨: 新形式に切替、`PackageScripts` も同一ファイル内で再 export する形で並存）
- 既存 user は `npx ts-git-hooks sync` を再実行することで新 d.ts 形式に migrate する。生成済み `git-hooks.d.ts` は上書き（型のみのファイルなので影響限定的）

### 決定理由

- `defineConfig()` wrapper 単独（Stage 1 のみ）では「結局 generic を書く必要がある」ため ergonomics 改善が限定的で、PR の宣伝価値が薄い
- module augmentation 単独（Stage 2 のみ）では `defineConfig` wrapper が存在しないため、user は依然として `export const config: TSGitHookConfig = { ... }` を書く必要がある（autocomplete は augmentation で効くが、`export default` という業界慣行に揃わない）
- 両者を同時に出すことで「`defineConfig({ ... })` 1 行で型が完結」というストーリーが成立する
- 後方互換を保つ実装にすることで、既存 user に migration を強制しない

### 却下した選択肢

| 選択肢 | 却下理由 |
|--------|----------|
| Stage 1 のみ実装 | generic 手書き問題が残るので「TypeScript First の ergonomics 改善」として弱い。同じ PR で出す cost が低いので分割する利益が薄い |
| Stage 2 のみ実装 | `defineConfig` を提供しないと `export default` 化できない。vite/vitest 風の業界慣行に合わない |
| 後方互換を切って `export default defineConfig` のみに統一 | breaking change になり、major version bump が必要。漸進的移行の機会を捨てる |
| `git-hooks.d.ts` を廃止して TS plugin で動的解決 | 実装コスト高、複雑度高。ROI 低い |
| `KnownScripts` を generic で渡す形式に留める（`defineConfig<KnownScripts>(...)`）| 結局 user が type 名を import する必要があり、ergonomics 改善が中途半端 |

### 影響範囲

変更されるファイル:
- `src/index.ts` — `defineConfig`, `KnownScripts` interface を追加 export
- `src/core/config.ts` — `loadConfig` で `default` export も受理（呼び出し元: `commands/install.ts`, `commands/list.ts`, `commands/verify.ts`, runner 内）
- `src/core/type-generator.ts` — 出力形式を declare merge 形式に変更
- `src/commands/init.ts` — default template を `defineConfig` 形式に変更
- `src/types.ts` — 既存 export は保持。`TSGitHookConfig` の default generic を `ResolvedScripts` にするか検討（互換性影響を要評価）
- 既存 test 群（`src/core/config.test.ts`, `src/core/type-generator.test.ts`, `src/commands/init.test.ts`）— 期待値の更新
- README.md / README.ja.md — 新しい推奨形式の追記、旧形式は backward-compat note として残す

### リスクレベル

**HIGH**

判定根拠: `loadConfig` は複数 command（install / list / verify / sync および generated hook の `run`）から呼ばれており、後方互換に失敗すると全 hook 実行が壊れる。境界線上のため上位 (HIGH) に寄せた。
