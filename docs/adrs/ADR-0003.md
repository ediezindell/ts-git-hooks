## ADR-0003: HookOptions を scope 別に分割

**日時**: 2026-05-03T00:00:00+09:00
**タスク概要**: `HookOptions` の two-tier 表現に起因する silent-no-op バグ（per-hook `replayFormatter` が型は通るが runtime で読まれない）を、`HookOptions` を `GlobalHookOptions` と `PerHookOptions` の 2 つに分割することで型レベルで解消する。後方互換は切る。

### コンテキスト

`HookOptions` は現在、top-level の `TSGitHookConfig` と per-hook の `HookConfigWithOpts` の両方に extend されているが、3 層が不整合:

| Option | TS 型 (per-hook) | Valibot (per-hook) | Runtime (per-hook) |
|---|---|---|---|
| `sequential` | ✅ 受理 | ✅ 受理 | ✅ 読む |
| `replayFormatter` | ✅ 受理 | ❌ warn（strict object で reject） | ❌ 読まない |

つまり user が
```ts
'pre-commit': {
  replayFormatter: true,  // TS は通る、valibot は warn、runtime は no-op
  config: { '*.ts': 'lint' },
}
```
と書いた場合、3 層が全部違うことを言う。`replayFormatter` の doc 自体が「Only takes effect for `pre-commit`」と謳っているにもかかわらず、その pre-commit の per-hook 設定では効かないという矛盾。

E2E テスト (`replay-formatter.spec.ts:74`) は top-level しか流していないので CI でこの乖離は検出されない。

### 決定内容

1. **`HookOptions` を 2 つに分割（`types.ts`）**:
   ```ts
   export interface GlobalHookOptions {
     sequential?: boolean;
     replayFormatter?: boolean;
   }

   export interface PerHookOptions {
     sequential?: boolean;
   }
   ```
   `HookOptions` は **削除**（alias を残さない）
2. **`HookConfigWithOpts` の base を `PerHookOptions` に変更**:
   ```ts
   export interface HookConfigWithOpts<_T extends string, C> extends PerHookOptions {
     config: C;
   }
   ```
3. **`TSGitHookConfig` の base を `GlobalHookOptions` に変更**
4. **Valibot schema (`config.ts`) を整合させる**:
   - 現在の `HookValueSchema` の wrapped 形は `{ sequential, config }` のみで `replayFormatter` を持っていないので、strict-object として既に reject 済み（warn が出る形）
   - ただし `replayFormatter` を明示的に「per-hook では受け付けない」と読めるよう、schema 構造を明確化（コメント追記またはキー列挙の明示）
5. **`runner.ts` は変更不要**: `replayFormatter` は元々 top-level しか読んでいない
6. **後方互換性なし**:
   - `HookOptions` の direct import は壊れる
   - `'pre-commit': { replayFormatter: true, config: {...} }` を書いていた user は型エラー（runtime 挙動は元から no-op だったので「無効だったコードを書けなくなる」変化）
   - CHANGELOG / release notes での明示が必要

### 決定理由

- 型・valibot・runtime の 3 層乖離を type system 側で消すことで、将来の option 追加時に同じ事故が起きる構造的余地を絶てる
- 後方互換を切ることで `HookOptions` alias の保守コスト・JSDoc deprecation 文言・将来削除タイミング判断が消える。alias を残すメリットが薄い
- option C（pre-commit-only option を型で表現する）は将来 pre-commit-only option がもう 1 個出てきた時点で考えれば足りる（YAGNI）

### 却下した選択肢

| 選択肢 | 却下理由 |
|--------|----------|
| A: per-hook `replayFormatter` を runtime で読むよう wire（最小修正） | silent-no-op を patch するだけで two-tier 構造はそのまま。将来の option 追加で同じ pattern を再生産する |
| C: `pre-commit-only` option を型で表現（`PreCommitOptions`） | `TSGitHookConfig` の mapped type が pre-commit / non-pre-commit の分岐に加えて option scope の分岐も持つことになり複雑化。pre-commit-only option がもう 1 個出るまでは過剰設計 |
| `HookOptions` を deprecated alias として残す | user 指示で却下。後方互換不要 |
| 現状維持 + doc comment で「per-hook 指定は無視」と注記 | type system が嘘をついている状態が残る。user は doc を読まない |

### 影響範囲

変更されるファイル:
- `src/types.ts` — `HookOptions` 削除、`GlobalHookOptions` / `PerHookOptions` 新設、`HookConfigWithOpts` / `TSGitHookConfig` の base 切替
- `src/core/config.ts` — `HookValueSchema` の意図を明確化（実体変更は最小限、現状の strict-object 挙動を維持）
- `src/core/config.test.ts` — per-hook `replayFormatter` が warn される test を追加
- `src/e2e/replay-formatter.spec.ts` — 変更不要
- README.md / README.ja.md — 変更不要（`HookOptions` を README が直接参照していない）
- runner.ts — **変更不要**

公開 API (`d.ts`) への影響:
- `HookOptions` の export は **削除** (breaking)
- `GlobalHookOptions`, `PerHookOptions` を新規 export
- `HookConfigWithOpts<_T, C>` の base interface が変わる

### リスクレベル

**HIGH**

判定根拠: `types.ts` は public API（published `.d.ts`）。`HookOptions` を削除する明示的な breaking change で、既存 user で直接 import している人がいれば確実に compile が壊れる。境界 (MEDIUM/HIGH) のため上位に寄せた。
