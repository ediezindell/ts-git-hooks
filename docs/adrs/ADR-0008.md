## ADR-0008: argsFn 用に quote helper を package から export

**日時**: 2026-05-03T17:30:00+09:00
**タスク概要**: セキュリティ Tier 1 として、`argsFn` (= `Command<T>` の tuple 第 2 要素) が shell operator (`&&`, `||`, `;`, `|`, redirects) を含む文字列を返した時に file 名が unquoted で shell に流れる問題を、ユーザが明示的に対処できるよう `quote(parts: string[]): string` helper を package から export する。runtime 挙動は変えず additive な API 追加。

### コンテキスト

`runner.ts:84-101` の `processCommand`:

```ts
if (isCommandTuple(command)) {
    const [script, formatArguments] = command;
    const result = formatArguments(files, script);
    const { args, hasOperators } = parseCommandLine(result);

    if (hasOperators) {
        return result; // <-- そのまま shell:true 経路へ
    }
    ...
}
```

string 形式 (operator 含む) では `${commandString} ${quote(files)}` で auto-quote されているのに対し、tuple+operator 経路は user の argsFn 出力を信頼するしかなく、対称性が崩れている。

具体的な攻撃可能性:

```ts
'*.ts': ['lint', (files) => `lint ${files.join(' ')} && echo done`]
```

ここで `files` に `; rm -rf /` のような file が混じっていると、`lint normal.ts ; rm -rf / && echo done` が shell に渡り RCE。git の file 名制限はあるが (NUL/`/` 不可)、`;`, ` `, `$()`, バッククォート等は許容されるので現実的に exploit 余地がある。

ただし `argsFn` の中身を library 側が分解して quote することはできない (file paths と user-defined flags を判別不能)。よって user に明示的 quoting の手段を渡し、JSDoc / README で促す方針が最適。

### 決定内容

1. **`src/utils/shell.ts` を新規作成** し、shell-quote の薄い wrap を export:
   ```ts
   import { quote as shellQuote } from "shell-quote";
   export const quote = (parts: string[]): string => shellQuote(parts);
   ```
   - shell-quote の input 型 (`(string | { op: string })[]`) を `string[]` に narrow して、API の意図を「files / strings の quoting」に限定
   - shell-quote の挙動変化 (将来的な major) を吸収する余地

2. **`src/index.ts` から re-export**:
   ```ts
   export { quote } from "./utils/shell";
   ```

3. **`src/types.ts` の `ArgsFn` JSDoc にセキュリティ注記を追加**:
   - 型 signature は不変 (additive、後方互換)

4. **README.md (両言語) の "Advanced: Custom Argument Formatting" 節に helper 例を追加**:
   - 「shell operator を使う場合は `quote()` を呼ぶ」を 1 段落で追記

5. **`src/commands/init.ts` の `DEFAULT_CONFIG_CONTENT` にコメント例を追加** (任意、template の hint として):
   - operator なしの簡素な default は維持
   - コメントで「shell operator を使うときは `quote` を import」と一行触れる

6. **`src/utils/shell.test.ts` を新規作成** し、`quote` の薄い passthrough をテスト

### 決定理由

- runtime 挙動を変えないので既存ユーザの設定は無影響 (additive)
- `quote` を package 外に持ち出すことで「shell-quote を直接 import せよ」と user に強いない (deps の implementation detail を漏らさない)
- JSDoc は IDE で見える場所なので discoverability が高い
- 型 signature は変えない (`ArgsFn` を `(files, script, quote?) => string` に拡張する案も検討したが、第 3 引数は使われない場合がほとんどで signature noise になる)

### 却下した選択肢

| 選択肢 | 却下理由 |
|--------|----------|
| `ArgsFn` signature を `(files, script, quote) => string` に変更 | 殆どの argsFn で第 3 引数は不要。signature noise が増える |
| `processCommand` 内で argsFn 結果を解析し file 名らしき部分を auto-quote | file 名と user-defined flag を判別する一般的方法が無い |
| operator 検出時に runtime warning を出す | 真正な意図の operator 使用にも noise |
| operator 検出時に実行を refuse | 既存ユーザの config を一律壊す |
| `shell-quote.quote` を re-export 直 (wrap せず) | shell-quote の input 型 union が漏れ、`{ op: '|' }` のような shell-quote 独自 spec を渡せてしまう |

### 影響範囲

変更されるファイル:
- `src/utils/shell.ts` — 新規 (`quote` export)
- `src/utils/shell.test.ts` — 新規
- `src/index.ts` — 1 行 export 追加
- `src/types.ts` — `ArgsFn` JSDoc 拡張 (型 signature 不変)
- `src/commands/init.ts` — `DEFAULT_CONFIG_CONTENT` のコメントに 1 行 hint
- `README.md` / `README.ja.md` — "Advanced: Custom Argument Formatting" 節に 1 段落追加

公開 API (`d.ts`) への影響:
- `quote: (parts: string[]) => string` を新規 export
- `ArgsFn` 型 signature 不変、JSDoc が拡張されるのみ
- 既存 export 不変

副次効果:
- runtime 挙動無変更
- shell-quote の dep は元々入っているので bundle size 増減なし
