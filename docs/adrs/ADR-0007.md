## ADR-0007: stashCreate の checkout-index を unstaged-only に絞らない (測定により却下)

**日時**: 2026-05-03T17:00:00+09:00
**タスク概要**: 速度改善 D 候補として `utils/git.ts:122-128` の `git checkout-index -f -a` を「`git diff --name-only -z` で取った unstaged-modified file list に絞った `checkout-index --stdin`」に置き換える案を検討。マイクロベンチで両者が同等のため却下。本 ADR は測定結果と却下判断を記録する。

### コンテキスト

`stashCreate` は `git stash create` で stash object を作った後、`checkout-index -f -a` で working tree を index と一致させ、tracked ファイルの unstaged 変更を消している。

仮説: 「`-a` は index 全件 (N) を walk するので、N が大きい monorepo (10k+ tracked) では I/O コストが効いてくる。unstaged-modified だけに絞れば O(K) (K << N) に削減できる」。

### 決定内容

**現状維持** (`checkout-index -f -a` を使い続ける)。

### 決定理由

合成 monorepo を組んでマイクロベンチで実測。`unstaged-modified = 50` を固定し、tracked 総数を 5000 / 10000 で計測:

| Tracked files | `checkout-index -f -a` | `diff --name-only -z \| checkout-index -f -z --stdin` |
|---:|---:|---:|
| 5,000 | ~20ms (3 試行とも) | ~20ms |
| 10,000 | ~30ms (3 試行とも) | ~30ms |

Git は `-a` でも index entry の stat 比較で up-to-date file を I/O から除外していると推測される。少なくとも我々が叩く粒度で「N に比例する I/O コスト」は観測できなかった。

提案案 (file-list 化) はむしろ:
- git invocation が 2 回 (`diff` + `checkout-index`) になり sys time が微増
- pipe を介すぶん shell 起動が 1 回挟まる (本実装は `execGit` で個別 spawn なので、ここは Node 側で list を渡す形になり pipe は不要だが、それでも 2 回 spawn)
- file 数が多いと OS の arg 長制限に当たる懸念があり、別途 stdin 経由にするなど実装が膨らむ

短期的にも長期的にも ROI が無い。

### 却下した選択肢

| 選択肢 | 却下理由 |
|--------|----------|
| `diff --name-only -z` + `checkout-index -f --stdin -z` | 上記実測で win なし。実装複雑化のみ |
| `git read-tree --reset -u HEAD` | `-a` と同等の挙動でメリットなし |
| 何もしない (= 採用) | これが本 ADR の決定 |
